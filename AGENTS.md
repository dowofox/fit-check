# NAES Agent Rules

이 파일은 Codex가 NAES 프로젝트에서 항상 따라야 하는 작업 규칙이다.

## 작업 전

- 최신 `origin/main`을 확인한다.
- `PRODUCT.md`를 읽는다.
- `ROADMAP.md`를 읽는다.
- 분류 관련 작업이면 `CLASSIFICATION_ISSUES.md`를 읽는다.
- 이미 구현된 기능인지 확인한다.
- 실제 최신 파일 내용을 기준으로 판단한다.

## 작업 선택

- 작업은 `옷 등록`, `옷장 관리`, `코디 추천`, `핏·사이즈 추천`, `사용자 경험·데이터 안전성`의 다섯 영역에서 선택한다.
- 긴급 P0가 없으면 `ROADMAP.md`의 `다음 실행 순서`에서 첫 미완료 작업을 선택한다.
- 동일한 핵심 영역의 작업을 두 번 연속 선택하지 않는다.
- 제품의 두 핵심 가치인 오늘의 코디 추천 또는 실측 기반 핏 설명에 직접 기여하는지 먼저 확인한다.
- 크래시, 데이터 손실, 명백한 잘못된 추천만 진행 순서를 중단하는 긴급 P0로 승격한다.
- 단일 상품명·키워드 예외는 `CLASSIFICATION_ISSUES.md`에 기록하고 반복성과 공통 원인이 확인되기 전에는 구현 작업으로 선택하지 않는다.

## 작업 원칙

- 한 번에 하나의 작업만 진행한다.
- 작업 범위가 너무 추상적이거나 결과가 이상해질 가능성이 있으면 구현 전에 사용자에게 다시 묻는다.
- 기존 구조를 최대한 유지한다.
- 기존 기능을 삭제하지 않는다.
- 불필요한 타입을 추가하지 않는다.
- 불필요한 state를 추가하지 않는다.
- 기존 UI를 유지하며 개선한다.
- 기능을 만들기 전에 화면을 더 단순하게 만들 수 있는지 먼저 검토한다.

## UI 원칙

사용자는 AI 분석을 보고 싶은 것이 아니다.

사용자는 결과를 보고 싶다.

따라서 가능하면 AI 내부 데이터는 숨기고 결과만 보여준다.

기본 UI에 보여줄 수 있는 것:

- 추천 사이즈
- 추천 코디
- 실측 기반 설명
- 사용자가 취할 수 있는 다음 행동

기본 UI에서 숨겨야 하는 것:

- confidence
- visualWeight
- pointLevel
- 추천 점수
- AI 분석 점수
- 내부 breakdown

## 추천 원칙

- 추천 점수는 내부 계산용이다.
- 추천 점수는 UI에 표시하지 않는다.
- breakdown은 UI에 표시하지 않는다.
- confidence는 UI에 표시하지 않는다.
- 사용자는 점수가 아니라 이유와 결과를 본다.

## 상품 등록 원칙

항상 상품 링크 등록을 우선한다.

상품 링크 등록은 공식 상품 정보, 공식 소재, 공식 실측을 가져와 추천 품질을 높이는 기본 흐름이다.

사진 등록은 빠른 등록 역할만 수행한다.

직접 등록은 링크나 사진이 어려운 경우를 위한 보조 흐름이다.

## 테스트

항상 다음 명령을 실행한다.

```bash
npm run lint
npx tsc --noEmit
```

추천 로직을 수정한 작업에서는 다음 회귀 테스트도 실행한다.

```bash
npm run test:outfit
```

상품 링크 추출을 수정한 작업에서는 다음 회귀 테스트도 실행한다.

```bash
npm run test:product-link
```

홈 로딩·추천 캐시·revision 저장을 수정한 작업에서는 다음 테스트도 실행한다.

```bash
npm run test:home-performance
```

데이터 백업·복원 형식이나 저장 흐름을 수정한 작업에서는 다음 테스트도 실행한다.

```bash
npm run test:data-backup
```

## Git

항상 다음 순서를 따른다.

1. 작업 완료
2. lint/typecheck 실행
3. 로컬 커밋 생성
4. 커밋 SHA 보고
5. 검증이 모두 통과하면 `origin/main`에 자동 push
6. push 결과 보고

다음 상황에서는 자동 push하지 않고 사용자에게 먼저 보고한다.

- lint 실패
- typecheck 실패
- 테스트 또는 검증을 실행하지 못함
- 작업 범위가 모호함
- 기존 기능 삭제나 큰 구조 변경이 필요함
- 충돌, push 실패, 원격 상태 불일치 발생

## Expo

Expo 관련 코드를 수정해야 할 때는 현재 프로젝트의 Expo 버전에 맞는 문서를 확인한다.

기존 Development Build 흐름을 깨뜨리지 않는다.

Expo 네이티브 모듈을 추가하거나 버전을 바꿀 때는 `npx expo install`을 사용하고,
`package.json`과 lockfile을 함께 갱신한 뒤 `npx expo-doctor`를 실행한다. 작업 완료
보고에는 네이티브 의존성 변경 여부와 새 Development Build 또는 EAS build가 필요한지
반드시 명시한다.

## 작업 종료 후

작업이 끝나면 반드시 다음 내용을 보고한다.

- 무엇을 수정했는지
- 왜 그렇게 수정했는지
- PRODUCT.md와 어떻게 연결되는지
- ROADMAP에서 완료한 항목
- lint/typecheck 결과
- 로컬 커밋 SHA
- push 결과
- 다음으로 추천하는 작업 1개
