# NAES Roadmap

## 제품 기준

로드맵은 다음 두 가지 핵심 가치에 직접 기여하는 작업을 우선한다.

1. 내 옷으로 오늘 입을 코디를 추천한다.
2. 새 옷이 내 몸에 어떻게 맞는지 상품 실측을 기준으로 설명한다.

분석 항목을 늘리는 것보다 사용자가 더 좋은 추천을 받고, 결과를 이해하고, 자신의 데이터를 안전하게 관리하는지를 먼저 본다.

## 작업 선택 규칙

1. 긴급 P0가 있으면 가장 먼저 처리한다.
2. 긴급 P0가 없으면 아래 `다음 실행 순서`의 첫 미완료 작업을 선택한다.
3. 한 번에 하나의 작업만 구현한다.
4. 동일한 핵심 영역의 작업을 두 번 연속 선택하지 않는다.
5. 작업을 마치면 완료 상태와 `마지막 완료 영역`을 갱신한다.
6. 다음 작업이 핵심 가치에 직접 기여하는지 구현 전에 확인한다.
7. 단일 상품명·키워드 예외는 `CLASSIFICATION_ISSUES.md`에 기록하고 곧바로 제품 작업으로 승격하지 않는다.

### 긴급 P0 기준

다음 중 하나에 해당할 때만 진행 중인 순서를 중단하고 P0로 즉시 승격한다.

- 앱 크래시 또는 핵심 흐름 진입 불가
- 저장 데이터 손실·덮어쓰기·복구 불가 위험
- 사용자가 실제로 입거나 구매할 판단을 크게 그르칠 명백한 잘못된 추천

단일 상품의 분류 누락, 새 키워드 별칭, 낮은 영향의 표시 문제는 긴급 P0가 아니다. 반복성과 제품 영향을 확인한 뒤 해당 핵심 영역의 일반 작업으로 묶는다.

## 현재 안정화 트랙

- [x] **날씨 단계별 진단과 실패 상태 정규화** · 사용자 경험·데이터 안전성
  - 권한 확인·요청, 위치 서비스, 현재 위치, API 요청·파싱, 캐시 읽기·쓰기를 개인정보 없이 단계별 측정한다.
  - 날씨 없음, 건너뜀, 실패를 분리하고 실패 원인을 홈 백그라운드 갱신 로그까지 전달한다.
- [x] **날씨 요청 캐시와 중복 방지** · 사용자 경험·데이터 안전성
  - 최근 위치와 저장 날씨를 우선 사용하고 같은 세션의 동시 요청은 하나의 in-flight Promise를 공유한다.
  - 위치 및 API 제한시간 이후에도 기존 추천과 유효한 날씨 캐시를 유지한다.
- [x] **홈 추천 인덱스의 실측 요약 분리** · 코디 추천
  - 코디 추천에 필요한 선택 사이즈 실루엣 요약만 보존하고 전체 `productSizeGuide` 구조는 상세·핏 화면에 남긴다.
  - 15·50·100개 옷장 fixture에서 직렬화 크기와 추천 결과 동등성을 함께 검증한다.
- [x] **추천 인덱스 버전 마이그레이션 강화** · 사용자 경험·데이터 안전성
  - 이전 인덱스는 전체 옷장에서 한 번만 재생성하고 다음 실행부터 캐시를 사용한다.
  - 손상된 인덱스나 생성 실패가 홈 진입과 기존 옷장 데이터를 막지 않도록 회귀 테스트를 추가한다.

## 다음 실행 순서

긴급 P0가 새로 확인되지 않는 한 아래 순서를 유지한다.

1. [x] **추천 피드백** · 코디 추천
   - 사용자가 추천 코디에 `좋아요` 또는 `덜 추천` 피드백을 남길 수 있게 한다.
   - 피드백은 다음 추천 순위에 반영하되 내부 점수나 분석값은 화면에 노출하지 않는다.
   - 전체 아이템 조합별 피드백을 저장하며 같은 선택을 다시 누르면 해제할 수 있다.
2. [x] **오늘 입었어요** · 옷장 관리
   - 사용자가 실제로 입은 코디를 명시적으로 기록한다.
   - 코디 저장과 착용 기록을 구분하고, 사용자의 행동이 있을 때만 착용 이력을 갱신한다.
   - 같은 코디는 하루 한 번만 기록하며 저장만으로 착용 이력을 변경하지 않는다.
3. [x] **데이터 백업·복원** · 사용자 경험·데이터 안전성
   - 옷장, 프로필, 저장 코디, 피드백과 착용 기록을 내보내고 복원할 수 있게 한다.
   - 복원 전에 데이터 형식을 검증하고 기존 데이터를 덮어쓸지 명확히 확인한다.

마지막 완료 영역: **사용자 경험·데이터 안전성**

## 핵심 영역별 미완료 작업

### 옷 등록

- [x] 분석 이미지 메모리·요청 크기 제한
  - 옷 등록과 코디 분석이 같은 이미지 인코더에서 응답 형식과 15MB 상한을 확인한다.
  - 원격 파일 크기를 알 수 있으면 Blob 생성 전에, 그 외에는 실제 Blob 기준으로 과대 요청을 차단한다.

- [x] 사진 일괄 등록의 중복 선택 제거
  - 앨범 결과의 빈 URI와 동일 URI를 선택 순서를 유지한 채 한 번만 분석한다.
  - 앱에서도 최대 10장 제한을 다시 적용해 플랫폼별 선택 결과 차이를 흡수한다.

- [x] 분석 이미지 응답 형식 검증
  - 상품 이미지 URL이 오류 상태나 HTML·JSON 응답을 반환하면 이를 base64 옷 사진으로 AI에 전송하지 않는다.
  - 빈 이미지와 잘못된 콘텐츠는 분석 실패로 처리하고 확인된 상품 정보 또는 기존 재시도 안내를 유지한다.

- [x] 사진 일괄 등록 선택 수 제한
  - 앨범의 무제한 다중 선택을 한 번에 10장으로 제한해 순차 AI 분석 요청이 과도하게 쌓이지 않게 한다.
  - 단일 사진·촬영·상품 링크 등록과 한 장 실패 후 나머지를 계속 처리하는 기존 동작은 유지한다.

- [x] 직접 등록 다시 선택 후 폼 복구
  - 직접 입력 폼에서 등록 방식 선택으로 돌아간 뒤 같은 직접 등록 카드를 다시 누르면 빈 입력 폼을 복원한다.
  - 링크·사진 등록의 현재 선택과 분석 초기화 동작은 유지한다.

- [x] 니트 집업의 편직 구조 우선 분류
  - 상품명·공식 카테고리에 니트 또는 편직 근거가 있으면 면 혼용률과 무관하게 일반 집업보다 구체적으로 분류한다.
  - 후드·스웨트·저지 집업은 니트 근거가 없으면 기존 일반 집업 분류를 유지한다.

- [x] 등록 필수값 검증 경로 통일
  - 화면의 최종 종류·색상·계절과 저장 직전 검증이 같은 정규화 결과를 사용한다.
  - 공식 상품 정보의 선택 항목이 비어 있어도 화면에 확정된 필수값이 있으면 등록을 막지 않는다.

- [x] 상품 URL 변경 시 이전 추출 결과 무효화
  - 상품 정보를 가져온 뒤 다른 URL을 입력하면 이전 상품의 이미지·공식 정보·분석 폼을 함께 비운다.
  - 입력창의 주소와 실제 저장되는 확정 상품이 달라지는 상황을 막고 새 링크를 다시 확인하도록 한다.

- [x] 등록 분석·저장 연속 실행 방지
  - 단일 분석, 여러 장 일괄 저장과 최종 옷 저장은 렌더 전 즉시 잠기는 공통 작업 경계를 사용한다.
  - 빠른 연속 탭으로 같은 사진이 서로 다른 ID의 옷 두 개로 저장되거나 분석 요청이 겹치지 않게 한다.

- [x] 오래된 상품 링크 응답 상태 덮어쓰기 방지
  - 링크 요청 중 URL이나 등록 방식을 바꾸면 이전 응답을 화면에 적용하지 않는다.
  - 사진 분석·일괄 저장 중에는 등록 방식을 바꿔 진행 중 데이터가 섞이지 않게 한다.

- [x] 상품 링크 대표 이미지 로컬 fallback
  - 상품 링크 이미지 다운로드에 성공하면 앱 문서 폴더에 원본 fallback을 보관한다.
  - 쇼핑몰 다운로드가 실패해도 링크 등록은 막지 않고 기존 원격 이미지로 저장한다.

- [x] 사진 등록 원본의 영구 저장
  - 앨범과 촬영에서 받은 임시 로컬 이미지를 옷장 저장 전에 앱 문서 폴더로 복사한다.
  - 상품 링크의 원격 이미지는 유지하고, 로컬 원본 복사에 실패한 사진은 저장 성공으로 처리하지 않는다.

- [x] 활성 등록 방식 재선택 시 입력 보존
  - 현재 선택된 링크·사진·직접 등록 카드를 다시 눌러도 입력 상태를 초기화하지 않는다.
  - 실제 등록 방식이 바뀔 때만 이전 방식의 이미지와 분석 결과를 정리한다.

- [x] 등록 항목 ID 충돌 방지
  - 단일·일괄 등록 모두 시각과 임의값을 조합한 공통 ID 생성기를 사용한다.
  - 같은 밀리초에 등록이 겹쳐도 기존 항목 ID와 충돌해 저장 완료를 오인하지 않게 한다.

- [x] 상품 링크 내부 네트워크 접근 차단
  - 상품 URL과 모든 리다이렉트 목적지의 DNS 주소를 확인해 로컬·사설 네트워크 요청을 막는다.
  - 공개 쇼핑몰과 무신사 단축 링크는 유지하면서 안전하지 않은 주소는 서버 요청 전에 거부한다.

- [x] 등록 API 무한 대기 방지
  - 사진 분석과 상품 링크 추출 요청에 용도별 제한 시간을 적용한다.
  - 네트워크가 응답하지 않아도 로딩 상태가 끝나고 기존 실패 안내로 복구되게 한다.

- [x] 빌드 환경별 API 주소 설정
  - 소스 수정 없이 `EXPO_PUBLIC_API_BASE_URL`로 등록·분석 서버를 교체한다.
  - 환경변수가 없으면 기존 로컬 개발 서버를 사용하고 중복 슬래시를 방지한다.

- [x] 동시 옷장 변경 직렬화
  - 여러 등록과 상세 수정, 삭제, 착용 기록이 겹쳐도 하나의 원본을 순서대로 갱신한다.
  - 동시에 완료된 추가와 서로 다른 필드 수정이 마지막 쓰기에 의해 유실되지 않게 한다.

- [x] 미입력 사이즈 저장 정규화
  - 화면의 `사이즈 미입력` 안내 문구를 실제 `ClosetItem.size` 값으로 저장하지 않는다.
  - 기존 데이터의 안내 문구도 선택된 사이즈로 오해하지 않고 핏 추천 문구에서 제외한다.

- [x] 상품 추출 실패 안내 통일
  - 링크 등록과 상세 상품 확정에서 같은 서버 오류를 같은 사용자 행동으로 안내한다.
  - 잘못된 링크, 미지원 쇼핑몰, 연결 실패를 구분하고 직접 입력 fallback을 유지한다.
- [x] 상품 링크 요청 전 검증
  - 링크 등록과 상세 상품 확정에서 같은 URL 정규화·프로토콜 검증을 사용한다.
  - 잘못된 주소는 서버 요청 전에 안내하고, 스킴이 생략된 상품 도메인은 안전하게 보완한다.
- [x] 등록 결과 확인 동선 단순화
  - 링크·사진·직접 등록 후 사용자가 반드시 확인해야 하는 종류, 색상, 계절만 짧게 보여준다.
  - 공식 정보와 추정 정보의 차이는 내부 우선순위로 처리하고 화면에는 필요한 행동만 안내한다.
- [x] 지원 쇼핑몰과 추출 상태 안내 정리
  - 공식 상품 정보·소재·실측의 지원 범위를 사용자가 이해할 수 있는 결과 문구로 정리한다.
  - 개별 키워드 예외가 아니라 여러 상품에서 반복되는 추출 실패 유형을 기준으로 개선한다.

### 옷장 관리

- [x] 사라진 세부 분류 필터 자동 복구
  - 선택한 세부 분류의 마지막 옷이 수정·삭제되면 유효하지 않은 선택을 `전체`로 복구한다.
  - 옷은 남아 있는데 오래된 필터 때문에 빈 옷장처럼 보이는 상황을 막는다.

- [x] 대규모 옷장 카드의 초기 렌더 제한
  - 검색·필터·정렬 결과를 처음 30개만 렌더링해 모든 이미지가 한 번에 마운트되지 않게 한다.
  - 기존 3열 구성과 전체 결과 수를 유지하고 사용자가 30개씩 이어서 확인할 수 있게 한다.

- [x] 옷장 필터·검색·정렬 반복 계산 제거
  - 옷장 원본이나 선택 조건이 바뀔 때만 카테고리 분류, 세부 필터, 검색과 날짜 정렬을 다시 계산한다.
  - 화면 폭이나 삭제 확인 같은 무관한 상태 변경에서는 기존 파생 목록을 재사용한다.

- [x] 옷장 그리드 화면 폭 변경 대응
  - 앱 시작 시 고정한 카드 폭 대신 현재 화면 폭으로 3열 카드 크기를 다시 계산한다.
  - 회전·분할 화면에서도 이미지 정사각 비율과 기존 간격을 유지한다.

- [x] 저장 코디 연결 확인 실패 시 옷 삭제 차단
  - 옷 삭제 전에 저장 코디 원본을 정상적으로 읽은 경우에만 연결된 코디 수를 안내한다.
  - 저장소 오류를 연결된 코디 없음으로 오인해 고아 코디를 만드는 삭제를 진행하지 않는다.

- [x] 사이즈 추천 순위의 반올림 동점 제거
  - 사용자에게 보여주는 정수 점수와 내부 정밀 순위 점수를 분리한다.
  - 근소하게 더 잘 맞는 사이즈가 상품 실측표의 행 순서 때문에 뒤로 밀리지 않게 한다.

  - 민소매는 어깨선이 명확하지 않은 상품 특성을 반영해 총장과 가슴 단면으로 사이즈 후보를 비교한다.
  - 사용자 어깨너비와 상품 어깨 실측이 없어도 추천을 막지 않으며 일반 상의와 긴팔 기준은 유지한다.

  - 서로 다른 저장 영역의 변경이 마지막 쓰기에 의해 사라져 오래된 홈 추천 캐시가 재사용되지 않게 한다.
- [x] 백업 스냅샷·복원과 앱 데이터 변경 직렬화
  - 백업 스냅샷은 진행 중인 옷장·프로필·저장 코디·피드백 변경이 끝난 뒤 한 시점의 데이터를 읽는다.
  - 복원 최종 쓰기도 같은 변경 경계에서 실행해 원본 데이터와 추천 revision이 서로 다른 시점으로 갈라지지 않게 한다.
- [x] 부분 손상 데이터의 불완전한 백업 차단
  - 옷장·저장 코디·피드백·착용 기록 배열에 잘못된 행이 섞여 있으면 유효한 행만 조용히 내보내지 않는다.
  - 일반 저장 변경과 같은 엄격한 검증을 통과한 데이터만 백업 파일 원본으로 사용한다.
- [x] 불완전한 백업 생성 방지
  - 백업 원본은 한 번의 엄격한 저장소 읽기로 가져오고 읽기 실패를 빈 데이터로 대체하지 않는다.
  - 깨진 JSON이나 잘못된 루트 형식이 있으면 불완전한 백업 파일 생성을 중단한다.
- [x] 저장 코디 변경 실패 격리
  - 삭제·이름 수정 실패를 정상적인 빈 목록과 구분해 기존 화면 데이터를 유지한다.
  - 실제 저장이 완료된 경우에만 저장 코디 목록을 갱신한다.
- [x] 데이터 백업·복원
  - `다음 실행 순서` 3번 작업과 동일하다.
- [x] 홈 추천 영구 캐시와 날씨 중복 계산 제거
  - 추천 아이템 ID 중심의 영구 캐시를 복원하고 캐시 날씨와 실시간 날씨가 의미 있게 다를 때만 다시 계산한다.
  - 삭제된 옷이나 오래된 revision이 포함된 캐시는 안전하게 무효화한다.
- [x] 홈 포커스 재로드 최소화
  - 데이터 revision이 바뀌지 않은 홈 재진입에서는 전체 저장소 재로드를 생략하고 날씨만 갱신한다.
- [x] 추천 조합 공통 준비 계산 제거
  - 추천 결과를 바꾸지 않고 액세서리 옵션 생성, 핵심 코디 정렬과 저장 조합 비교의 반복 비용을 줄인다.
- [x] 저장 데이터 마이그레이션 검증
  - 새 optional 필드가 추가되어도 기존 옷장과 프로필이 깨지지 않는지 버전별로 확인한다.
- [x] 손상된 로컬 저장 데이터 격리
  - 배열이 아닌 저장값과 필수 필드가 깨진 항목을 읽기 단계에서 제외해 화면 크래시를 막는다.
- [x] 복구 가능한 오류 처리
  - 네트워크·분석 실패 시 입력한 정보가 사라지지 않고 다시 시도하거나 수동 등록으로 이어지게 한다.

## 분류 예외 처리

상품명, 소재명, 카테고리 별칭처럼 개별 분류 사례는 `CLASSIFICATION_ISSUES.md`에서 관리한다.

다음 조건을 충족하기 전에는 이 문서의 실행 큐에 넣지 않는다.

- 여러 상품 또는 쇼핑몰에서 반복된다.
- 공통 규칙으로 해결할 수 있다.
- 코디 추천이나 실측 기반 핏 설명에 의미 있는 영향을 준다.

## 완료 이력

기존 `P0 N차` 완료 목록은 Git 이력에 보존한다. ROADMAP은 완료 작업의 장기 보관소가 아니라 현재 제품 우선순위와 미완료 작업을 관리하는 문서로 사용한다.

## 완료 기준

- 기존 기능과 저장 데이터 호환성을 유지한다.
- 사용자 화면은 내부 분석보다 결과와 다음 행동을 중심으로 보여준다.
- 관련 회귀 테스트를 실행한다.
- `npm run lint`를 통과한다.
- `npx tsc --noEmit`을 통과한다.
- 검증이 모두 통과하면 로컬 커밋을 만들고 정해진 Git 절차에 따라 반영한다.
